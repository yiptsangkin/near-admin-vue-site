# 构建发布

# 构建

near-admin的构建相对来说比较灵活，可以支持不同环境的构建，同时也还支持单独构建工程中的某个单页应用，发布应用的时候不再需要整体打包。从package.json中 我们可以看到在本地运行和在构建方面near-admin都提供了比较丰富的指令。

{
     "serve": "vue-cli-service serve",
     "serve:translate": "node script/translate.js && vue-cli-service serve",
     "translate": "node script/translate.js",
     "create:page": "node script/newpage",
     "build": "vue-cli-service build",
     "uat:build": "cross-env BUILD_ENV=uat vue-cli-service build",
     "prod:build": "cross-env BUILD_ENV=prod vue-cli-service build",
     "analyze:build": "cross-env ANALYZE=true vue-cli-service build",
     "single:build": "node script/singleoperate.js build",
     "single:serve": "node script/singleoperate.js serve"
}

其中我们常用的指令是npm run serve，这个指令可以在本地运行整个工程，即工程中auth、manage等单页应用将会同时构建编译，如果我们有的时候不想要 在本地同时运行两个单页应用，即可通过npm run single:serve来选择你要运行的单页应用，加快编译的速度。

? Pick target entry ›  
◉   auth
◯   manage

例如这里我选择了只编译auth这个单页应用，那么我们从浏览器里面就只能访问/auth/login，访问/manage则显示空白。

同理，构建也支持单独构建某一个单页应用，这样就可以在发布的时候只更新某一个单页应用的内容。

提示

需要注意的是，在使用npm run single:build构建的时候，需要选择是否清空dist文件夹，如果你是想要增量发布某一个单页，而不影响之前发布的其他单页应用的内容， 请选择不要清空，这样直接整个替换dist就可以了。

# 环境选择

正如前面mock中提到的，构建不同的环境，我们只需要在build/config.js中增加对应的环境，并且到package.json中增加相应的脚本即可。near-admin默认 了dev uat prod这三个环境，所以，在package.json中增加了一下三个命令：

{
    "build": "vue-cli-service build",
    "uat:build": "cross-env BUILD_ENV=uat vue-cli-service build",
    "prod:build": "cross-env BUILD_ENV=prod vue-cli-service build"
}

可以看到其实就是按照BUILD_ENV这个参数对打包环境进行区分，由于windows和unix等环境命令行不太一致，因此需要通过cross-env的插件来实现命令行的兼容， 所以如果需要增加环境，除了在build/config.js中增加相应的环境外，还需要在这里加多一个指令，例如，我们要增加一个sit的环境，就这么写：

{
    "sit:build": "cross-env BUILD_ENV=sit vue-cli-service build"
}

# 分析构建体积

一般来说，通过near-admin这种架构，构建出来的文件一般不会存在单文件比较大的情况，再加上gzip的压缩，基本上是可以忽略的白屏体验，但是框架还是提供了 利用构建分析插件来帮助大家进行构建优化。

npm run analyze:build

# 服务端发布

构建完成后，输出的文档就在根目录的dist文件夹中，理论上我们把里面的文件随便丢到一个web服务器就大功告成了，但是这里面以下三点需要注意。

提示

作者推荐使用nginx作为web服务，其他的web服务应用也同理进行处理就可以了（因为电脑没有搞其他的，所以例子就用nginx的）。

# 1、路由模式的选择

vue-router的路由支持两种模式，一种是history，一种是默认hash，near-admin官方推荐是使用history模式，当然，如果你喜欢#号这种的话也可以，修改 方面只需要到对应单页的router里面修改对应想要的mode即可。

// src/pages/auth/router/index.ts
const routerOpt: RouterOptions = {
    mode: 'history' //  modify here
}

# 2、html后缀处理

修改完成后，由于是多页模式，因此在一般的web服务配置下，是需要带上.html后缀才能够访问的，例如：

http://localhost/auth.html

因此我们需要修改nginx的配置，将其修改为支持不带html后缀访问。(大家可以在本地安装nginx进行测试)

server {
    listen       8004;
    server_name  localhost;
    root /Users/administrator/Documents/project/open-source/near-admin-vue/dist;

    location / {
        try_files $uri $uri/ $uri.html?$args;
    }
}

这里我配置了8004端口的服务，然后根目录是我们构建出来的地址，这里使用了try_files兼容了不带html后缀的访问，运行nginx，在浏览器中访问

http://localhost:8004/auth

发布

可以看到我们能够成功访问对应的前端页面，但是有一个问题，就是如果你进行刷新操作，你会发现页面将无法进行访问，这是因为nginx没办法访问到auth/login和auth/login.html 这两个地址，因此会提示500错误，主要是因为使用了后端服务的路由。所以遇到这个情况，有两种解决方式，首先我们先看第一种，就是使用hash的模式，我们修改下工程中项目对应的 router，改为hash模式并重新打包。构建完成后我们重新访问。

http://localhost:8004/auth

你会发现访问地址变成了

http://localhost:8004/auth#/login

而且进行刷新并没出现错误的现象，这是因为vue-router通过hash模式进入了前端的路由控制。如果不追求地址栏的洁癖，那么你的发布就此完成，可以进行使用啦！

# 3、history模式下刷新问题

如果你追求地址栏没有#号的话，则可以在nginx中增加兼容history的配置。

server {
    listen       8004;
    server_name  localhost;
    root /Users/administrator/Documents/project/open-source/near-admin-vue/dist;

    location ~/(.*?)(/|$) {
        port_in_redirect off;
        index /$1.html;
        try_files $uri /$1.html;
    }
}

这里通过修改匹配到地址的第一项内容，即单页应用的应用名来识别访问哪一个html文件，加上这个配置之后，我们重新访问history模式构建出来的包，可以看到， 即使刷新也能够正常访问，这样，应用的发布就大功告成啦！

提示

作者本人强烈推荐用history模式吧，如果使用的是hash模式的话，可能auth里面的一些跳转开发者们需要注意下自行进行修改
