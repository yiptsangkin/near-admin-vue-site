# Mock

# 本地mock

near-admin使用better-mock（基于mockjs）实现了纯前端的本地mock，他的原理很简单，就是通过匹配你请求中的相关的路径信息，然后返回你提前设定好的数据。正如网络请求那一节提到， 你可以通过isMock这个开关来控制是否请求mock数据。near-admin的mock的数据单独在根目录下的mock文件夹中维护。其中，index.ts用于维护匹配规则， 而data中则用于mock数据。

// src/mock/index.ts
import comConfig from '@custom/config'
import mockData from 'better-mock'
import {getSendMsg, checkLoginByAccount, checkLoginByPhone, getUserMenu, getUserInfo} from '@mock/data/common'

if (comConfig.buildSwitch.isMock) {
    // here is rules
    mockData.mock(/\/mock-data\/getSendMsg/, getSendMsg)
    mockData.mock(/\/mock-data\/checkLoginByAccount/, checkLoginByAccount)
    mockData.mock(/\/mock-data\/checkLoginByPhone/, checkLoginByPhone)
    mockData.mock(/\/mock-data\/getUserMenu/, getUserMenu)
    mockData.mock(/\/mock-data\/getUserInfo/, getUserInfo)
}

export default mockData

mock数据方面，框架层面也有规范约束，需要根据ComRespone来编写你返回的mock数据。

import {ComRespone} from '@corets/type'

export const getSendMsg = (req: any): ComRespone => {
    console.log(
        `%c near-admin %c here mock request %c`,
        'background:#35495e ; padding: 1px; border-radius: 3px 0 0 3px;  color: #fff',
        'background:#dbaf05 ; padding: 1px; border-radius: 0 3px 3px 0;  color: #fff',
        'background:transparent'
    )
    console.log(`%c${JSON.stringify(req, null, 4)}`, 'color: #dbaf05')
    return {
        code: 0,
        message: '获取成功',
        data: {
            msgCode: (1000 + parseInt((Math.random() * 8999).toString(), 10)).toString()
        },
        timestamp: new Date().getTime()
    }
}

其中返回值中的console部分是框架推荐的约定，也可以不用加，加了的话在调试的过程中可以将请求参数展示出来。

# 外部mock

在实际开发中，使用本地mock相对来说比较少，也有另外的方案是基于mock-server的，但是作者本人任务完全没有必要去使用mock-server，直接使用外部的一些 包含mock+接口文档的工具即可，例如Yapi。这类型的mock就更加简单了，使用devServer直接代理到他们的mock地址即可。

# 未来展望

请求这块作者还有很多的想法，例如直接根据mock数据生成对应typescript的数据类型，然后在mock中完善返回数据的类型，让开发使用起来更加的方便，还有就是 收集日常一些接口错误返回然后自动生成相关的mock测试用例，不过这块需要比较多的设计时间和开发时间，希望有朋友也能加入到开发进来。
