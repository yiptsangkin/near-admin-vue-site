# 服务端交互

# 前端请求流程

在near-admin中，一个完整的前端UI交互到服务端处理流程是这样的：

  • 在assets/ts/api中创建对应前端模块的文件目录，并创建apiUrl.ts以及apiMethod.ts两个文件；
  • 在apiUrl.ts中维护请求的地址，注意，需要将host部分定义为读取配置文件；
  • 在apiMethod.ts中封装请求事件，使用到工具库utils.ts中sendReq方法对请求事件进行编写；
  • 选择async/await或者回调的请求模式；
  • 获取请求响应；

接下来我们通过auth应用中的登录事件这个简单的例子来说明下如何按照约定编写一个网络请求。

# 网络请求

1、首先我们在assets/ts/api中创建了auth目录用于存放登录模块用到的相关接口，并创建apiUrl.ts以及apiMethod.ts分别存放地址以及请求方法。

2、我们编辑apiUrl，在里面写入一个根据用户账户密码登录的接口。这里需要注意的是，apiHost统一通过comConfig中的apiHost属性来维护，在near-admin中， 会通过apiConfigProd.js，apiConfigUat.js，apiConfigDev.js这三个独立文件来维护api请求目标服务，方便一些通过docker等容器技术动态生成文件， 而不是打包到代码本身当中。

提示

near-admin打包配置已经通过判断打包的环境默认在打包出来的html模板上外部引入了相关的apiConfig文件。

// apiUrl.ts
import comConfig from '@custom/config';

const apiHost = comConfig.apiHost

export default {
    checkLoginByAccount: `${apiHost}/mock-data/checkLoginByAccount`
}

3、在apiMethod.ts中编写具体的请求逻辑，需要注意的是，请求参数需要按照ReqType的类型进行提交。

export interface ReqType {
    headers?: any,
    responseType?: | 'arraybuffer' | 'blob' | 'document' | 'json' | 'text' | 'stream',
    method?: string,
    timeout?: number,
    data?: any,
    url?: string,
    success?: any,
    fail?: any,
    error?: any
}

// apiMethod.ts
import apiUrl from './apiUrl'
import utils from '@corets/utils'
import {ReqType, ResType} from '@corets/type'

const checkLoginByAccount = (reqType: ReqType) => {
    reqType.method = 'POST'
    reqType.url = apiUrl.checkLoginByAccount
    const result = utils.sendReq(reqType)
    return result
}

export default {
    checkLoginByAccount
}

4、在对应的前端页面中引入相应的apiMethod，并进行网络请求方法的调用。对于near-admin来说，提供的请求库是支持async/await以及回调两种模式的，开发者 可以根据自己的需要习惯进行使用，区别在于使用回调方法的话，框架本身会对请求方式结果进行初步的判断，主要是根据http的状态码和返回json的code值来区分。而通过 async/await则需要自己进行请求结果进行判断。

import api from '@api/auth/apiMethod'

...

// 回调模式
api.checkLoginByAccount({
    data: {
        username,
        password
    },
    success: (res: any) => {
        self.$message.success(
            self.$t(dict.localeObj.loginForm.loginSuccess) as string,
            2
        ).then(() => {
            self.toManage()
        })
    },
    fail: (res: any) => {
        self.$message.error(res.message)
    }
})

# 多环境目标服务控制

正如上面提到的，near-admin的目标服务配置不会打包到代码当中，支持动态生成，对于本地开发来说，建议都通过devServer进行代理，这样子不用担心打包的时候 忘记修改请求地址，只需要选择正确的环境参数即可。

如果框架内置的三个环境prod uat dev不满足开发者使用的话，可以自行到build/config中的envList中增加环境，这样子在构建的时候，框架会自动在 public/static/js中创建对应的配置文件。

// build/config.js
module.exports = {
    envList: [
        'dev',
        'uat',
        'prod'
    ]
}

提示

需要注意的是，如果自行增加了对应的环境，需要到package.json中增加对应环境的打包指令。

# 从Mock与服务相互切换

框架层面在配置中增加了一个isMock的开关，如果打开的话默认会走匹配mock的规则，如果关闭的话则相当于不会拦截请求直接请求对应的目标服务。开发者可以在 assets/ts/custom/config中修改defaultSwitch的isMock属性，或者直接在管理界面的个性化设置那里进行调整也可以。

// assets/ts/custom/config.ts
const defaultSwitch = {
    isMock: true,
    isHotKey: true,
    isBreadCrumb: true,
    isCache: true,
    isI18n: true
}

# 网络请求本地缓存

对于有些数据量比较大的数据（如一些基础树状下拉数据，不经常变更），我们希望在请求时候不走后端请求或者不走后端的缓存（因为有一定的下载时间），我们想放在前端存储的话， 可以在请求的headers处设置local-cache: true，这样的话，同样参数和请求地址的网络请求将会直接请求本地存储localStorage去获取返回值。

提示

需要注意的是，localStorage本身有大小的存储，因此这个本地缓存使用需要比较小心，不过也不用过于担心，框架层面已经处理好一些超存储的逻辑，一旦localStorage满了将会 清空一部分的缓存数据。

// assets/ts/api/manage/apiMethod.ts
const getUserMenu = (reqType: ReqType) => {
    reqType.method = 'POST'
    reqType.headers = {
        'local-cache': true
    }
    reqType.url = apiUrl.getUserMenu
    const result = utils.sendReq(reqType)
    return result
}
