# 路由和菜单

# 路由

正如前面介绍所说，near-admin是一个基于多页多路由的中后端开发框架，因此，路由的配置方面也是跟着不同单页应用目录，例如auth和manage下面都有自己的router配置文件， 具体配置的内容跟vue-router官方的配置没什么太大的区别，但是需要注意的是在manage下的router需要增加一部分缓存router的配置，用于标签独立页动态生成组件。

// 缓存代码片段
const routesList: string = localStorage.getItem(`${dict.commonObj.managePath}AsyncRoute`) || ''
let newRoutesList: CacheRouteConfig[]
try {
    newRoutesList = JSON.parse(routesList)
    if (!Array.isArray(newRoutesList)) {
        newRoutesList = []
    }
} catch (e) {
    newRoutesList = []
}

newRoutesList.forEach((item, index) => {
    item.component = () => import(`../view/${item.componentPath}`)
})

if (routerOpt && routerOpt.routes) {
    routerOpt.routes = routerOpt.routes.concat(newRoutesList)
}

near-admin内置了router的缓存路由的类型文件，所以如果开发者想要动态缓存某个路由的话，可以按照类型字段要求缓存在localstorage本地存储中。

# 菜单

near-admin内置了菜单项（页面组件）的类型文件，具体如下所示

export interface CpInfo {
    component: string,                               // 组件目录地址
    title: string,                                   // 标签title
    navIndex?: string,                               // 导航序号(菜单打开才有)
    params?: CpParams,                               // 菜单参数
    isUrl?: boolean,                                 // 是否是url
    pk?: string | number | boolean,                  // 主键
    breadList?: any[]                                // 面包屑列表
}

首先我们看下CpInfo这个类型，其中component字段为对应的页面组件在manage/views中的相对路径，例如， 我想要打开homepage/HomePage.vue这个页面组件，那么我的component字段就是homepage/HomePage， title为打开标签页的标题，即在标签栏上面显示的文本，navIndex主要是侧边菜单的id，这个字段用于高亮当前选中的字段， params的话则是这个标签页带的参数，默认带的参数为CpParams中的类型，开发者也可以根据自己需要的字段进行穿参， isUrl是一个隐藏字段来的，框架本身就会根据cpmponent是否为url进行判断，pk为这个标签页的主键，缓存部分也是按照这个主键进行缓存的，这个也是框架内部自己处理的 breadList如果是通过菜单点击的话会默认按照菜单的结构生成面包屑列表，也支持用户自行配置。

interface CpParams {
    apiNew?: boolean,                                // 是否通过api创建
    withoutCache?: boolean,                          // 是否允许缓存
    isAffix?: boolean,                               // 是否固定标签
    checkSave?: boolean,                             // 关闭标签是否提示保存
    dataUrl?: string                                 // url(外链)
}

接下来我们看下菜单项的默认参数的含义，首先是apiNew，这个是用于判断是否由页面api创建的标签页，这类型的标签页将不进行权限判断，withoutCache这个字段是用于判断这个 标签页是否进行缓存，isAffix这个字段则是判断是否在标签栏固定这个标签（固定的标签将不会被关闭），checkSave这个字段是用于判断这个页面是否在关闭的时候进行操作确认 ，dataUrl则是用于存储外链的链接，这个一般由框架进行处理。

# 嵌套菜单

项目中模拟了菜单结构的数据，菜单的第一层级属于菜单分组，分组将会在Header那里进行展示，点击可以切换到不同的菜单组，而菜单组下具体的菜单都是按照内置的类型文件中的NavList，侧边菜单最多支持三层。

export interface NavList {
    name: string,                      // 标签名，对应CpInfo中的title
    path?: string,                     // 路径，对应CpInfo中的path
    icon?: string | {
        value: string,
        type: 'common' | 'aicon'
    },                                 // 图标，aicon代表ant-design内置的icon，common则指的是自己引入的iconfont中的图标
    groupName?: string,                // 菜单分组
    params?: any,                      // params参数
    child?: NavList[]                  // 自菜单
}

# 面包屑

项目在每个组件的对象中都使用了breadList作为面包屑列表的存储，用户点击菜单的话，框架本身会自动将菜单的层级进行提取自动生成面包屑列表，而对于api生成的页面则不进行修改，如果想要修改的话开发者 可以根据类型文件的字段将面包屑列表存储到对应的组件状态中。面包屑组件还支持下拉功能，切换组件信手拈来。

# 外链

目前大部分的中后台项目都仅支持新开一个浏览器标签页进行外链的打开，near-admin支持在框架内标签新开一个外链。

# 侧边栏默认展开

由于near-admin很多属性都是通过全局状态进行控制的，因此侧边菜单这边对于子菜单的展开也是通过全局状态来维护，这里是通过src/store/modules/global.ts中的defaultExpandKeys来控制默认展开哪些菜单，key的内容 为menu组件中的key，具体怎么找可以在dom上面搜索nav-index，对应的值便是菜单的key。
